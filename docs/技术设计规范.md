# DK日常开销 —— 技术设计规范

## 1. 技术栈

| 层次 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | 原生 HTML5 + CSS3 + JavaScript | 无需构建工具，零依赖 |
| UI 方案 | CSS Flexbox + Grid | 移动端适配 |
| 图表库 | Chart.js 4.x | 轻量图表方案 |
| 数据存储 | Web Storage API (localStorage) | 持久化本地存储 |
| PWA | Service Worker + manifest.json | 离线缓存 + 添加桌面 |
| 字体 | 系统字体栈 | -apple-system, PingFang SC |

## 2. 架构设计

### 2.1 整体架构

```
index.html          ← 入口页面（单页应用）
├── css/style.css   ← 全部样式
├── js/app.js       ← 全部逻辑
├── manifest.json   ← PWA 配置
├── sw.js           ← Service Worker
└── icons.svg       ← 应用图标
```

### 2.2 设计模式

采用 **模块化单例模式**，所有状态集中管理：

```
state = {
  year: number,        // 当前查看年份
  month: number,       // 当前查看月份
  records: Array,      // 全部记录
  settings: object,    // 设置
  currentPage: string  // 当前页面
}
```

### 2.3 页面路由

单页应用，三个页面通过 JS 控制显示/隐藏：

| 页面 | ID | Tab索引 |
|------|----|---------|
| 首页（账单） | #page-home | 0 |
| 统计 | #page-stats | 1 |
| 设置 | #page-settings | 2 |

新增记录页为全屏覆盖层（`#add-overlay`），从底部滑入。

## 3. 数据模型

### 3.1 记录对象

```javascript
{
  id: string,           // 唯一ID (时间戳+随机数)
  amount: number,       // 金额
  category: string,     // 大类名称
  subCategory: string,  // 子类名称
  date: string,         // 日期 "YYYY-MM-DD"
  note: string,         // 备注
  createdAt: number     // 创建时间戳
}
```

### 3.2 设置对象

```javascript
{
  monthlyBudget: number  // 月预算，默认5000
}
```

### 3.3 存储键名

| 键名 | 存储内容 |
|------|----------|
| `dk_records` | 记录数组 JSON |
| `dk_settings` | 设置对象 JSON |

## 4. 编码规范

### 4.1 命名规范

- **变量/函数**：驼峰命名，如 `getMonthRecords`
- **常量**：全大写，如 `CATEGORIES`、`CHART_COLORS`
- **CSS 类名**：连字符命名，如 `.record-item`、`.category-grid`
- **ID 选择器**：连字符命名，如 `#add-overlay`

### 4.2 DOM 操作

- 使用 `$()` 和 `$$()` 简写替代 `querySelector`
- 所有 DOM 引用集中在 `dom` 对象中
- 事件绑定统一在 `bindEvents()` 函数中管理

### 4.3 数据安全

- 所有 `JSON.parse` 操作包在 try-catch 中
- 金额展示统一用 `formatMoney()` 保留两位小数
- 输入验证：金额上限、小数位数限制

---

## 5. 兼容性要求

- 最低支持 iOS 14.5（2021年发布）
- Safari 浏览器，启用 Service Worker 和 Web Storage
- 适配安全区域（safe-area-inset-bottom）
- 禁用 iOS 上自动缩放和长按菜单

---

## 6. PWA 离线策略

- Service Worker 在 `install` 阶段预缓存所有静态资源
- 采用 **Cache First** 策略：优先从缓存读取，回退到网络
- 每次更新 Service Worker 时清理旧缓存
